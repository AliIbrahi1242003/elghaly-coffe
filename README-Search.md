# README-Search.md

## نظرة عامة على وحدة البحث والترشيح
في هذا المشروع، البحث عن المنتجات يتم بطريقتين متكاملتين:
1. **بحث فوري داخل واجهة المستخدم (Search Overlay في Navbar)**: يجلب قائمة مختصرة من المنتجات عبر API ثم يرشّحها على العميل بناءً على نص البحث.
2. **ترشيح صفحة المتجر**: عند الانتقال إلى `/shop?search=...` تقوم صفحة المتجر بتحميل المنتجات كلها عبر Server Action ثم تُرشّحها محليًا بناءً على نص البحث.

هذا يعني أن البحث الحالي **ليس بحثًا في قاعدة البيانات مباشرة**، بل هو **فلترة على العميل** لبيانات تم جلبها مسبقًا.

---

## الملفات والدوال المرتبطة بوحدة البحث

| الملف | الدور |
| --- | --- |
| `app/api/search/products/route.ts` | API endpoint يعيد بيانات مختصرة للبحث الفوري (id/title/category/description). |
| `components/Navbar.tsx` | واجهة البحث الفوري (Modal) + منطق جلب البيانات + فلترة الاقتراحات. |
| `app/shop/page.tsx` | فلترة المنتجات بناءً على `search` في الـ query string. |
| `app/actions/products.ts` | Server Action يجلب كل المنتجات مع العلاقات (صور/مقاسات/أنواع) لاستخدامها في صفحات المتجر والمنتج. |
| `lib/schema.ts` | تعريف جدول المنتجات (الحقول المستخدمة في البحث: title/category/description). |

### تفاصيل مختصرة لكل جزء
- **`app/api/search/products/route.ts`**:  
  يعيد قائمة منتجات **بدون صور أو أسعار** لتقليل الحمولة، فقط حقول البحث.
- **`Navbar.tsx`**:  
  - يفتح نافذة البحث.
  - عند فتحها لأول مرة يجلب البيانات من `/api/search/products`.
  - يفلتر النتائج محليًا ويعرض اقتراحات.
  - عند اختيار نتيجة يوجّه إلى `/shop?search=...`.
- **`app/shop/page.tsx`**:  
  يستقبل `search` من URL ويقوم بفلترة المنتجات بعد جلبها.
- **`app/actions/products.ts`**:  
  مصدر البيانات الأساسي للمنتجات في كل الصفحات.

## كيف يعمل البحث خطوة بخطوة

### 1) البحث الفوري في الـ Navbar
1. المستخدم يضغط زر البحث.
2. `Navbar` يستدعي `/api/search/products` إن لم تكن البيانات محمّلة.
3. يتم تخزين المنتجات في `searchProducts`.
4. كلما كتب المستخدم نصًا:
   - يتم دمج `title + category + description`.
   - يُحوَّل النص إلى lowercase ويُقارن بنص البحث.
5. تظهر نتائج فورية (حد أقصى 5).

### 2) الانتقال لصفحة المتجر
1. عند اختيار اقتراح أو إرسال النموذج:
   يتم الانتقال إلى `/shop?search=...`.
2. في `app/shop/page.tsx`:
   - يتم استدعاء `getProducts()`.
   - يتم فلترة المنتجات بنفس المنطق (text includes search term).
3. النتائج تُعرض كـ Product Cards.

## كيف تفهم الوحدة بسهولة (Mentor Path)

1. **ابدأ من الـ UI**  
   اقرأ `components/Navbar.tsx` وركّز على:
   - `isSearchOpen`
   - `searchProducts`
   - `handleSearchSubmit`
   - `matchingProducts`

2. **افهم مصدر البيانات**  
   راجع `app/api/search/products/route.ts` لتعرف الحقول التي تُستخدم في البحث الفوري.

3. **افهم فلترة المتجر**  
   اقرأ `app/shop/page.tsx` واكتشف كيف يُعاد استخدام نفس منطق الفلترة.

4. **اربط كل شيء بمصدر المنتجات**  
   راجع `app/actions/products.ts` لمعرفة لماذا بيانات صفحة المتجر أوسع من بيانات البحث الفوري.

## كيف تعدّل أو توسّع وحدة البحث

### 1) إضافة حقول بحث جديدة
- أضف الحقل في:
  - `app/api/search/products/route.ts`
  - منطق الفلترة في `Navbar.tsx` و`app/shop/page.tsx`

### 2) جعل البحث يعتمد على قاعدة البيانات
بدل الفلترة في العميل:
- أنشئ API جديد يرسل `search` للسيرفر.
- نفّذ استعلام Drizzle مع شروط `LIKE` أو `ilike` (حسب محرك DB).
- واجه المستخدم بنتائج جاهزة بدل جلب كل المنتجات.

### 3) تحسين الأداء للكتالوج الكبير
- أضف Pagination في `app/shop/page.tsx`.
- لا تجلب كل المنتجات للفلترة.
- حافظ على نتائج البحث في الـ server.

### 4) دعم البحث بالعربية
- أضف حقول عربية (مثل `titleAr`) في المخطط والبيانات.
- عدّل منطق البحث لتضمين الحقول العربية.

---

## ملاحظات هندسية مهمة
- البحث الآن **Client-side فقط** → مناسب لعدد منتجات صغير/متوسط.
- البحث الفوري يستخدم **ذاكرة الواجهة** ولا يستدعي السيرفر لكل حرف.
- عند زيادة البيانات، يفضّل نقل البحث للسيرفر.

---

## أقول إيه في المناقشة عن شغل البحث
1. أنا حدّدت أن البحث عندنا بيشتغل بطريقتين: بحث فوري من الـ Navbar + فلترة في صفحة المتجر بعد الانتقال بـ query string.
2. شرحت إن البحث الحالي Client-side وليس بحثًا مباشرًا في قاعدة البيانات، وده مناسب للكتالوج الصغير/المتوسط.
3. حدّدت ملفات الوحدة الأساسية: `Navbar.tsx` لواجهة البحث، و`app/api/search/products/route.ts` للبيانات المختصرة، و`app/shop/page.tsx` للفلترة.
4. وضّحت مسار البيانات: API يُرجع حقول مختصرة، ثم يتم فلترتها في الواجهة وعرض الاقتراحات.
5. قدمت خطوات واضحة لتطوير الوحدة لاحقًا (نقل البحث للسيرفر، إضافة حقول عربية، وتحسين الأداء).
